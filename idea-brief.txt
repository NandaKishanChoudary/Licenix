# Smart Licensing Assistant — NitroStack Build Spec

Build a production-quality MCP server called **"Smart Licensing Assistant"** using the
**NitroStack** TypeScript framework (`@nitrostack/core`, `@nitrostack/cli`, optionally
`@nitrostack/widgets`) — not the bare `@modelcontextprotocol/sdk`.

The goal: simplify restaurant licensing in Kerala using AI + MCP. The MCP server is the
intelligence layer; the AI client (NitroChat, ChatGPT, Claude Desktop, or any
MCP-compatible client) is only the interface. The client should never answer licensing
questions from its own knowledge — it must invoke these tools.

---

## 0. Scaffold the project (do this first, don't hand-roll boilerplate)

```bash
npx @nitrostack/cli init smart-licensing
cd smart-licensing
npm install
npm run dev
```

Then open the project folder in **NitroStudio** (https://nitrostack.ai/studio) for
live tool testing, payload inspection, and chatting with the server as you build.

Requirements: Node.js >= 20.18, npm >= 9.

---

## 1. Scope (unchanged from original spec)

- Business: **Restaurant** only.
- Location: **Kerala** only.
- Any other business (Hospital, Hotel, School, Pharmacy, Manufacturing, Retail) or any
  other state must short-circuit at `business_analyzer` with:
  ```json
  { "supported": false, "message": "This business category is not supported in the current prototype." }
  ```
- Never fabricate licensing information. If live/cached government data isn't
  available, return an explicit "unavailable" field rather than a guess.

## 2. Project structure (NitroStack conventions)

```
smart-licensing/
  src/
    app.module.ts                  # @McpApp root — registers everything below
    main.ts                        # bootstrap via McpApplicationFactory
    tools/
      business.tools.ts            # 1. business_analyzer
      license.tools.ts             # 2. license_discovery
      regulation.tools.ts          # 3. regulation_lookup
      document.tools.ts            # 4. document_and_forms
      cost.tools.ts                # 5. cost_estimator
      timeline.tools.ts            # 6. timeline_estimator
      advisor.tools.ts             # 7. ai_advisor
      package.tools.ts             # 8. generate_licensing_package (orchestrator)
    services/
      business.service.ts          # @Injectable — validation + district/dept lookup
      license.service.ts           # @Injectable
      regulation.service.ts        # @Injectable
      document.service.ts          # @Injectable
      government-fetcher.service.ts # @Injectable — live fetch + fallback chain
      cache.service.ts             # file/memory cache used by government-fetcher
      link-validator.service.ts    # @Injectable — official-domain allowlist + HEAD check
      ai-advisor.service.ts        # @Injectable — LLM call or heuristic fallback
    data/
      restaurant-license-catalog.ts # stable taxonomy (licenses, docs, depts)
    widgets/
      app/
        licensing-guide/page.tsx   # optional rich UI for generate_licensing_package
    types/
      index.ts
  .env.example
  package.json                     # generated by CLI, don't hand-write
  tsconfig.json                    # generated by CLI, don't hand-write
```

Key differences from a hand-rolled `@modelcontextprotocol/sdk` build:
- Tools are **decorated methods on plain classes**, not `server.registerTool(...)` calls.
- Business logic lives in **`@Injectable()` services**, injected into tool classes via
  constructor DI — tool handlers stay thin (per NitroStack best practice).
- `app.module.ts` wires everything with `@McpApp` + `@Module({ providers, imports })`.

## 3. The 8 tools

Each tool below keeps the exact name, purpose, and I/O contract from the original spec —
only the implementation shape changes.

### 3.1 `business_analyzer`
```typescript
@Tool({
  name: 'business_analyzer',
  description: 'Validate the business type/location and identify industry, category, state, district, and responsible government departments.',
  inputSchema: z.object({
    business: z.string().describe('Business type, e.g. "restaurant"'),
    location: z.string().describe('City or district in Kerala, e.g. "Kochi"')
  }),
  annotations: { readOnlyHint: true, idempotentHint: true, destructiveHint: false, openWorldHint: false }
})
async analyze(input: {...}, ctx: ExecutionContext) {
  return this.businessService.analyze(input.business, input.location);
}
```
Returns `{ supported, businessCategory, state, district, departments }` or the
`supported: false` short-circuit shape.

### 3.2 `license_discovery`
Returns every required license (Trade License, FSSAI, GST, Fire NOC, Pollution
Consent, Health, Signage, Liquor if applicable) — each with `name`, `mandatory`,
`issuingDepartment`, `description`, `renewalFrequency`.

### 3.3 `regulation_lookup`
Returns applicable Acts/Rules: Food Safety Rules, Kerala Municipality regulations,
Fire Safety Regulations, compliance notes — structured, not prose.

### 3.4 `document_and_forms`
The most detail-heavy tool. Per license: required documents (Aadhaar, PAN, rental
agreement, building plan, ID/address proof), official form, verified PDF/apply-online
link (via `LinkValidatorService`), submission method (online/offline), department
contact info.

### 3.5 `cost_estimator`
```typescript
@Tool({ name: 'cost_estimator', description: '...' })
@Cache({ ttl: 86400, key: (input) => `cost:${input.business}:${input.district}` })
async estimate(input: {...}, ctx: ExecutionContext) {
  return this.costService.estimate(input);
}
```
Per-license application/inspection/renewal fee + total, sourced through
`GovernmentFetcherService`. Fields are `"unavailable"` rather than invented when the
source can't be confirmed. `@Cache` (NitroStack's built-in decorator) is a good fit
here since fees don't change often — set `ttl: 86400` (24h) to match the fetcher's own
cache policy.

### 3.6 `timeline_estimator`
Same shape as cost_estimator: application/inspection/approval timeline + overall
duration, same `@Cache({ ttl: 86400 })` treatment.

### 3.7 `ai_advisor`
```typescript
@Tool({ name: 'ai_advisor', description: '...' })
async advise(input: {...}, ctx: ExecutionContext) {
  return this.aiAdvisorService.recommend(input.previousToolOutputs);
}
```
`AiAdvisorService` calls Claude (via `ANTHROPIC_API_KEY` if set) or falls back to a
deterministic heuristic derived only from the other tools' structured output — never
invents facts. Produces: best application order, prep checklist, documents to prepare
first, common mistakes, warnings, expected completion time, tips for faster approval.

### 3.8 `generate_licensing_package` (orchestrator)
```typescript
@Tool({
  name: 'generate_licensing_package',
  description: 'Run the full licensing pipeline for a restaurant in Kerala and return one combined Licensing Guide.',
  inputSchema: z.object({ business: z.string(), location: z.string() }),
  invocation: { invoking: 'Assembling your licensing guide...', invoked: 'Licensing guide ready' },
  examples: { request: { business: 'restaurant', location: 'Kochi' }, response: { /* trimmed example */ } }
})
@Widget('licensing-guide')
async generate(input: {...}, ctx: ExecutionContext) {
  const analysis = await this.businessService.analyze(input.business, input.location);
  if (!analysis.supported) return analysis;

  const [licenses, regulations, documents, costs, timelines] = await Promise.all([
    this.licenseService.discover(analysis),
    this.regulationService.lookup(analysis),
    this.documentService.getForms(analysis),
    this.costService.estimate(analysis),
    this.timelineService.estimate(analysis)
  ]);
  const advice = await this.aiAdvisorService.recommend({ analysis, licenses, regulations, documents, costs, timelines });

  return { analysis, licenses, regulations, documents, costs, timelines, advice };
}
```
Important: the orchestrator calls the **services directly** (constructor-injected),
not the other tools — that's the DI pattern NitroStack expects, and it avoids
tool-calling-tool overhead/loops.

Optionally attach `@Widget('licensing-guide')` so NitroChat/NitroStudio render the
combined guide as a formatted UI component instead of raw JSON — this is one of the
concrete advantages of building on NitroStack instead of the bare SDK. If you skip
this, the tool still works fine and returns structured JSON.

## 4. Live government data — same fallback chain as before

`GovernmentFetcherService` (`@Injectable`) hits FSSAI / GST / Kerala LSG / Fire &
Rescue / KSPCB / Excise public pages:

**fresh cache → live fetch (retry w/ backoff) → stale cache → explicit "unavailable"**

- No CAPTCHA/auth bypass — public unauthenticated pages only.
- `LinkValidatorService` checks any returned URL against a `*.gov.in` / `*.nic.in`
  allowlist and probes it (HEAD/GET) before marking `verified: true`.
- This custom multi-tier fallback stays as its own service — NitroStack's `@Cache`
  decorator handles simple tool-response caching (used on `cost_estimator` /
  `timeline_estimator` above) but doesn't replace the fetcher's own retry/staleness
  logic, which is domain-specific.

---

## 5. Error handling & logging

- Throw plain `Error` (or custom `NotFoundError`/`ValidationError` classes) from
  service methods — NitroStack's tool runtime catches these and returns a structured
  MCP `isError: true` result automatically; you don't need to hand-wrap every handler.
- Logs go through `ctx.logger` (structured, stderr) inside tool handlers, or an
  injected logger in services — never `console.log`, since stdout is the JSON-RPC
  channel.
- Optionally add `@UseFilters(...)` with a custom exception filter if you want to
  reshape error responses uniformly across all 8 tools.

---

## 6. Design principles (carried over)

- One tool = one responsibility; business logic lives in services, not tool handlers.
- Zod schemas on every tool's `inputSchema` (and `outputSchema` where practical) with
  `.describe()` on every field.
- `readOnlyHint: true` / `openWorldHint: true` annotations set accurately per tool
  (all 8 tools here are read-only; several are open-world since they hit live gov
  sites).
- Architecture should allow adding Hospital/Hotel/School/etc. later without a rewrite:
  keep `restaurant-license-catalog.ts` swappable — add
  `{business}-license-catalog.ts` files and branch in `BusinessService.analyze()`.

## 7. Testing & running clients

- **NitroStudio**: open the project folder directly — it runs the dev server and
  gives you tool testing, payload inspection, and an integrated AI chat to try
  `generate_licensing_package` with `{"business":"restaurant","location":"Kochi"}`.
- **NitroChat / Claude Desktop / ChatGPT**: same stdio launch config
  (`command: node`, `args: ["dist/server.js"]` after `npm run build`, or point at the
  dev server per NitroStack's docs) — no code changes needed since tool logic is
  transport-agnostic.
- Type-check with `tsc --noEmit`, build with the CLI's `build` command, and smoke-test
  each service function directly before wiring into tools.

---

## 8. What NOT to do

- Don't call `@modelcontextprotocol/sdk`'s `McpServer`/`registerTool` directly — use
  NitroStack's `@Tool`/`@Module`/`@McpApp` decorators throughout; that's the whole
  point of this refined spec.
- Don't put fetch/caching/validation logic inline in a tool handler — it belongs in an
  `@Injectable()` service.
- Don't fabricate fee amounts, SLAs, or PDF URLs that haven't been confirmed reachable
  — return `"unavailable"` instead, exactly as in the original spec.
